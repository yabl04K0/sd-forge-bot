# ЧЕК — протокол глубокого аудита кода

Запускается командой `/chek`, `/chek <файл/модуль>`, или `/chek всё`.
Аргументы: `$ARGUMENTS`

- `/chek` — обычный прогон по всему проекту с учётом суппрессов.
- `/chek <файл>` — аудит только указанного файла/модуля.
- `/chek всё` — игнорировать суппрессы (`chek_never.md`, отложенные), сканировать заново всё.

## Реестры (в корне проекта, коммитятся вместе с кодом)

- `chek_open.md` — активные проблемы.
- `chek_never.md` — won't-fix навсегда. **Молча пропускать при сканировании.**
- `chek_later.md` — отложенные. Напоминать в футере каждого прогона.

**Идентичность проблемы:** `file::symbol::pattern` — НЕ номер строки (он дрейфует).
**Инвариант:** каждая проблема живёт ровно в одном реестре.

## Прогон по шагам

### 1. Загрузка реестра
Прочитать все три файла.
- `chek_never.md` / `chek_later.md` — **GC**: если символ исчез из кода → запись устарела, убрать.
- `chek_later.md` — если код вокруг символа изменился относительно `deferred_sha` → поднять запись в активные (`chek_open.md`).
- `/chek всё` — игнорировать суппрессы, сканировать заново всё.

### 2. Сканирование
Читать файлы проекта **полностью** (не grep). Искать баги трёх уровней:

- **КРИТИЧНО:** race conditions; дубли при retry / `timed_out`; молчащий `except`; утечки ресурсов без `finally`; путаница return-кодов.
- **ВЫСОКИЙ:** логические ошибки; неправильный результат; счётчики не обновляются на всех путях; файлы не чистятся при ошибке.
- **СРЕДНИЙ:** UX-мусор; потеря исключений из `create_task`; устаревший стейт.

### 3. Отчёт
Вывести список: `SEVERITY | file:line | описание`. Спросить «приступать?».

### 4. Фиксер
Минимальный фикс для каждой находки КРИТИЧНО/ВЫСОКИЙ. **Не рефакторить.**
Записать в `chek_open.md`, что сделано (или инкрементировать `passes_run`).

### 5. Критик
Независимая проверка: перечитать каждый изменённый файл заново и проверить:
- фикс устраняет причину, а не симптом;
- не сломан ли соседний код;
- есть ли необработанный edge case.

Нашёл → добавить `subfindings` в `chek_open.md`, вернуться к фиксеру (шаг 4).
Если `passes_run >= 3` без схождения → эскалация, сообщить человеку.

### 6. Regression-тесты
Для каждой закрытой КРИТИЧНО/ВЫСОКИЙ — минимальный тест, который провалился бы до фикса.
Запустить все тесты.

### 7. Обновление реестров
- Закрытая проблема → удалить из `chek_open.md` (вечная запись = коммит + тест).
- Отложено → перенести в `chek_later.md` с `deferred_sha`.
- Инвариант: проблема живёт ровно в одном файле.

### 8. Футер
Вывести в конце прогона:

```
ЧЕК завершён.
  активных: <N>   закрыто за прогон: <M>   отложено: <K>
  напоминания (chek_later): <список>
  passes_run: <P>
```

### 9. Коммит + вопрос «пушить?»
Закоммитить изменения с понятным сообщением, затем спросить «пушить?».

## Форматы записей

### chek_open.md
```yaml
- id: "file::symbol::pattern"
  severity: КРИТИЧНО | ВЫСОКИЙ | СРЕДНИЙ
  file: "path/to/file.py"
  symbol: "function_or_class"
  description: "что не так"
  fix: "что сделано (минимальный фикс)"
  passes_run: 0
  subfindings: []
  test: "путь/к/regression-тесту или null"
```

### chek_never.md
```yaml
- id: "file::symbol::pattern"
  reason: "почему won't-fix навсегда"
```

### chek_later.md
```yaml
- id: "file::symbol::pattern"
  reason: "почему отложено"
  deferred_sha: "<git sha кода вокруг символа на момент откладывания>"
```
